# 🧠 2. 핵심 로직 복기 & 새로 습득한 파이썬 무기
오늘은 '불필요한 인덱싱을 버리고 파이썬 다운 자료구조와 연산자 활용기' 를 중점으로 두고 학습하려고 했다.

    - 음수 슬라이싱 `[-4:]` : 슬라이싱 [시작:끝] 은 기본적으로 왼쪽에서 오른쪽(정방향)으로 진행되는데, 뒤에서 4번째 자리부터 끝까지 자르려면 역방향 인덱스 [-1:-4] 가 아니라 시작점만 지정한 [-4:] 를 지정했어야 했음

    - List Comprehension : [x for x in arr if 조건] 으로 필터링과 배열 생성을 한 줄로 압축해보려고 했다.
        - 파이썬에서는 빈 리스트는 False 로 평가됨을 알았고, 이를 활용해서 결과 or [-1] 형태로 if-else 예외처리를 압축 할 수 있었음.
        - 리스트 컴프리헨션 자체가 약간.. 아직은 어려움이 있는 것 같아서 많이 활용을 해봐야 할 것 같음

    - Hash(딕셔너리) 활용 : 탐색 문제에서는 `요소: 등장 횟수` 로 카운팅 하는 '상태 기록장' 형태로 쓰면서 key:value 를 탐색하는 조건으로 잘 활용해봐야 겠다는 생각이 들었음. (이부분도 아직 어렵다..)

# 🚨 3. 피드백 및 부족했던 점 (잘못된 접근과 원인)
객관적으로 파이썬 문법 자체에 어느정도 쓸 수 있는 함수들이랑 이런것들은 존재하는데, 코딩방식이 아직 C/JAVA 를 못벗어나는 상황인 것 같음.
    
    1. 가장 심각한 안티 패턴 : 인덱스 집착 ex) range(len(arr))
        - 접근 : 배열의 요소를 꺼낼 때 위에 예시처럼 인덱스를 억지로 생성하려고 하는게 문제임.
        - 원인 : 옛날 습관이 베여있어서 리스트 컴프리헨션을 써야하는 상황에서도 .append() 를 사용하면서 코드 줄이 더 길어지는 것 같음

    2. 반복문 제어 실패
        - 접근 : 아직도 for 문을 돌릴 때 문제 해석에 맞게 반복 범위를 잘 설정하지 못하는 것 같음
        - 원인 : '최소값' 을 찾는 문제 정의를 잘 봐야되거나.. 범위 설정에 대한 부분을 잘 인지해야 할 것 같음.

    3. 딕셔너리 용도 오해
        - 접근 : 딕셔너리를 단방향 매핑 (참가자:완주자) 로만 생각해서 매핑 과정에서 갑자기 뇌정지가 왔음.
        - 원인 : 빈도수를 센다는 개념을 적용하지 못해서 문제에 정의한 '동명이인' 케이스는 어떻게 할꺼야? 라는 문제에 접근조차 못했음.
