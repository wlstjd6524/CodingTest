# 🧠 2. 핵심 로직 복기 & 새로 습득한 파이썬 무기
오늘 학습의 핵심은 '사람의 논리' 나 'C언어 또는 Java 같은 구조' 를 버리고 파이썬 내장함수나 문법을 적극적으로 활용하면서
내장함수를 익히고 코드를 경량화 하는 것을 목표로 두고 진행하였다.

    - 프로그래머스 입문 문제 중 math.gcd(a, b) 최대공약수 : 분수의 덧셈 시 경우의 수를 나누어서 하나하나 if-else 구문으로 하드코딩 하는 방식 없이, import math 를 통해 math 에서 제공하는 gcd 함수를 이용하여 (분모1 * 분모2) 형태로 보편적 통분을 진행한 뒤 단번에 기약분수로 처리했다.

    - Boolean 반환의 간소화 : if A==B: return True 같은 형식은 파이썬에서는 안티 패턴임. 파이썬 에서는 애초에 A == B 에 따른 조건을 계산한 다음에 Return 값을 True, False 로 주기 때문에 여기서도 if-else 로 하나하나 하드코딩 할 필요가 없었음.

    - 리스트 컴프리헨션(중요) : [int(x) for x in str(n)[::-1]] 과 같은 리스트 컴프리헨션 타입을 사용하여 코드 경량화를 진행할 수 있었음.

    - 언패킹 : 크기가 고정된 2차원 배열에서 2중 for 문으로 접근하면서 인덱스를 하나하나 받아오는 패턴은 코드가 길어지기 때문에 그냥 for char in str: 에서 str 문자를 하나씩 받는 케이스 처럼 for i, j, k in commands: 형태로 곧바로 변수에 꼽아서 활용 할 수 있는 점을 인지하여 직관성을 높일 필요가 있었음.

# 🚨 3. 피드백 및 부족했던 점 (잘못된 접근과 원인)
백준 문제에서도 일단 무작정 코드 구현 조금 할 줄 알고, 로직 생각 좀 할 줄 안다고 커리큘럼을 쉽게쉽게 건너 뛰면서 완전탐색, BFS 로 막 넘어가려고 했었는데 실질적으로 다시 파이썬 내장함수를 쓰면서 코드경량화를 해보려고 하니까 아직까지는 파이썬 문법 자체에서 어려움이 존재하는 것 같았다.

이것들을 기반으로 다음 문제들이 존재한다고 느꼈다.
    
    1. 논리적 허점과 데드 코드 생성
        - or 과 and 로직을 문제와 다르게 정의하여 논리적 허점 발생.
        - 또한 이미 상단에 if A == B: 식으로 코드 접근이 이루어진 상태에서 문제 정의만 보고 없어도 되는 즉, 코드 기능은 똑같은데 문법처리만 다른 부분을 또 정의해서 데드코드를 생성해버림.

    2. for ~ in 문의 오해
        - 백준에서도 해당 부분에 대한 오류가 있었는데 for g in commands: 로 접근하고 거기서 인덱스를 하나하나 뽑아보려고 하는 인덱싱 에러를 범함.
        - 해당 케이스는 만약 commands 가 [2,3,4] 로 정의되어 있으면 하나씩 뽑아오는게 아니라 [2,3,4] 자체를 g에 저장해버리기 때문에 인덱싱을 할 수가 없었음.
